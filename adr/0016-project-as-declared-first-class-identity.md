# 0016 — Project as declared first-class identity

Under ADR 0009 every memory lives in `$HOME/.mom/mom.db`, which solved the multi-vault source-of-truth problem but left a different one open: a memory knows its `session_id` and its provenance, yet has no notion of which project it belongs to. Recall over this global pool degenerates quickly. A query like "native integration" run from a small project gets dominated by memories from large, vocabulary-dense projects whose terms happen to overlap; BM25 has no signal to prefer the caller's actual project. Tags carry incidental file-path crumbs from tool calls, not project identity. Path-based heuristics fail the moment two machines hold the same project at different locations; git-remote-based heuristics fail for non-git projects and require fragile URL canonicalization across renames, transfers, and forks.

A project is a first-class entity. Each project declares its identity in a checked-in file at the project root: `.mom-project.yaml`, version 1, containing at minimum a stable `id` string. The file is data the project owns — like `.editorconfig` or `tsconfig.json` — and travels with the repository across machines, clones, forks, and transfers. The retired project-local `.mom/` *data* directory is not coming back; this is a single configuration file at the project root, not a state directory. When the `/mom-project` skill writes this file, it stamps a watermark comment at the top making ownership explicit:

```yaml
# MOM project binding — owned by you, not regenerated.
# Edit freely; check this file into version control.
version: "1"
id: foo
```

`id` is treated as an opaque user-chosen string. The data layer validates only basic sanity (non-empty, no newlines, no NULL bytes, length-capped). The skill normalises interactively-supplied values (lowercase, trim, dashes for spaces) so the common case stays clean, but MOM does not enforce a strict slug format — if a team wants `id: My Service 🚀` and it works for them, that is their decision. MOM does not opine on how users organise projects, repos, or products.

Every memory row carries `project_id TEXT` (nullable). At capture time, the resolver walks up from the session's working directory looking for `.mom-project.yaml`; if found, the declared `id` is stamped on every memory the session produces; if not found, `project_id` is left NULL. Resolution canonicalises symlinks (per the `pathutil.CanonicalDir` invariant) and picks the longest registered ancestor when nested project files exist — MOM does not warn, block, or take a position on monorepo vs flat layouts. Resolution runs at turn-publish time with a short cache (~5s) rather than only at watcher start, so a user who runs `/mom-project` mid-session sees subsequent captures pick up the new id without restarting their session. The global watch registry at `~/.mom/watch-registry.json` is demoted from source-of-truth to a `(dir → id)` cache for daemon scheduling; the `.mom-project.yaml` file is authoritative.

Recall is project-scoped by default. `mom recall <query>` resolves the active project from cwd and filters; `--project=<id>` overrides the resolution explicitly; `--all` disables scoping. When cwd is not bound to any project, recall falls through to all-projects with a stderr hint pointing the user at the binding flow. Legacy memories captured before this ADR have `project_id = NULL` and remain findable in scoped queries by default; `--strict-project` excludes them when the user wants a clean view of post-binding memories only. `project_id` is a structural filter applied in `WHERE`, not a ranking signal; it does not enter `memories_fts`.

The CLI is the resolution surface. `mom record` and `mom recall` resolve `project_id` from cwd through a shared helper. User-facing MCP tools (`mom_recall`, `mom_record`) are slated for retirement on a separate track; while they exist, they accept `project_id` only as an explicit argument and never guess from the MCP server's startup cwd. The data-collection MCP surface (transcript ingestion via the watcher pipeline) is unaffected — `project_id` rides on the existing `turn.observed` payload.

Project binding is a deliberate user action, not a side effect of `mom init`. The global install flow (`mom init`) configures the vault, daemons, and harnesses; it does not write project files. Project binding is surfaced through a skill — `/mom-project` — that the LLM agent runs interactively when a user wants to bind the current directory. The skill proposes an `id` (slugified from the directory basename), confirms with the user, and shells to `mom project bind --id <chosen>` to write the YAML. The CLI primitive exists for scriptable contexts (CI, automation); the skill is the user-facing front door. The `mom-status` skill, which runs at session start, surfaces "this directory is not bound to a MOM project" as a one-line informational nudge — discoverability without silent auto-creation and without an active prompt to bind. Users in scratch directories, exploratory work, or one-off scripts can ignore the line; users who care notice it and act.

`mom uninstall` never deletes `.mom-project.yaml`. Path-1 (disconnect this project) and Path-2 (full uninstall) both leave the file in place. The user's binding decision is theirs, persists across reinstalls, and is typically checked into version control alongside other project metadata.

`project_id` on the memory row is substance-immutable per ADR 0011: written once at insert, never updated. The `id` value inside `.mom-project.yaml` is owned by the user — they can edit it directly or ask an agent to. MOM neither prevents nor reconciles such edits. The behaviour when an `id` changes is therefore deterministic and intentional: subsequent captures use the new `id`; previously-captured memories keep the old `id` on their rows; the two cohorts do not merge. Practically, changing an `id` is equivalent to starting a fresh project — the old memories remain reachable via `--project=<old-id>` or `--all`, but the project's recall scope from now on is the new `id`. No rename-and-merge command is shipped; if users need that, they can always query both ids with `--project=<old-id>,<new-id>` (multi-value support, a small extension we can ship later) or fall back to `--all`.

## Consequences

- Recall stops being dominated by the largest project. The user's current project wins by default; cross-project queries are explicit.
- The schema gains one column and one index. No new tables.
- The watch registry is a cache, not a source of truth. Rebuilding it from `.mom-project.yaml` files is straightforward.
- Cartographer v2, Cursor v2, team sync, Lens project views, and `/mom-reflect` all join on `project_id`. The contract is established before those features are built.
- Sync becomes a transport problem, not a schema problem. Alice and Bob both ship memories with `project_id: momhq-mom`; whatever transport delivers them (CRDT, server, S3) can merge cleanly without path translation or URL canonicalisation.
- Legacy memories from before this ADR remain reachable via scoped recall (NULL inclusion default) and via `--all`. They are second-class only when the user opts into `--strict-project`.
- Projects without a `.mom-project.yaml` keep working — sessions capture memories with `project_id = NULL`, the user gets a status nudge each session start, and they bind when they're ready.
- ADR 0011 (substance-immutability) is extended: `project_id` joins the substance set.

## Considered alternatives

- **Path-as-identity (canonical absolute path).** Rejected: breaks the moment two users hold the same project at different paths. Even for single-user multi-clone (e.g. `~/work/foo` and `~/scratch/foo`), the same project shows up as two recall pools.
- **Git-remote-as-identity (canonicalised origin URL).** Rejected: requires fragile URL canonicalisation (ssh/https, credentials, trailing `.git`, host case), excludes non-git projects entirely, and turns repository transfers/renames into history-orphaning events. Inferring identity from a remote is heuristic; declaring it is not.
- **Opaque numeric `project_id` minted by Librarian.** Rejected: not shareable. Alice's and Bob's machines mint different ids for the same project; sync would need a join table mapping local ids to a shared key, which is exactly the problem this ADR avoids by making the declared `id` the shared key.
- **Auto-create `.mom-project.yaml` during `mom init`.** Rejected: `mom init` is the global install flow, not the per-project binding flow. Running it per project would conflate two responsibilities and create awkward UX (re-running global init every time the user opens a new repository).
- **Silent auto-create on first capture.** Rejected: the `id` is a shared artefact (it gets committed and persists across users). It deserves a deliberate user choice. Auto-guess is the right behaviour for non-shared local state (e.g. cache paths); it is the wrong behaviour for cross-machine identity.
- **A `projects` table with `(id, dir, slug, name, members…)`.** Rejected for v1: adds a table, a migration, and write coordination for no current consumer. Re-evaluated when cartographer v2 needs project-level metadata beyond identity.
- **Strict `id` validation (slug regex enforced at the data layer).** Rejected: `id` is the user's identifier, not ours. We use it as a SQL column value and a comparison key; we do not use it as a filename, URL path segment, or anything that requires escaping. The skill normalises interactively-provided values for ergonomics; the data layer accepts whatever non-pathological string the user puts in their YAML.
- **Mandatory namespace convention (e.g. `vendor/project`, DNS-style).** Rejected for v1: namespacing is a sync-time concern, and sync does not yet exist. Constraining the format now would lock in assumptions about a transport we have not designed. A `namespace:` field can be added later additively when sync's needs are concrete.
- **Lock `ProjectId` once at watcher start (require restart after `mom project bind`).** Rejected: produces silent-NULL-after-bind surprise that generates confused bug reports. The cost of resolving at publish-time with a short cache is negligible and the UX is correct.
- **Active prompt at session start when cwd is unbound.** Rejected: intrusive for scratch / exploratory / one-off work. Encourages reflex-bind in directories that should not have a binding. An informational line is enough; users who want to bind invoke `/mom-project` directly.
- **Auto-delete `.mom-project.yaml` during `mom uninstall`.** Rejected: the file is user-owned data, typically version-controlled, and represents a deliberate choice. Removing it on uninstall and then reinstalling would force users to re-bind and risk picking a different id, creating exactly the merge-confusion this ADR is designed to avoid.
